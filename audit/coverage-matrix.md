# TeraSlab Test Coverage Matrix

HEAD: `1e5659b` (branch `main`) | Built by tracing `tests/` against `src/protocol/opcodes.rs`.

Legend: ✅ covered · ⚠️ partial (see note) · ❌ missing.

"Covered" means a test exercises that op **through the cited path** and asserts a
meaningful outcome (not merely that the op compiled). Engine-level (in-process
`Engine` API) coverage is distinguished from wire-level coverage
because a money-safety bug can live in the dispatch mapping layer that engine
tests never touch. Wire-level coverage comes in two flavors here:
- **TCP** (full socket round-trip) — `tests/*_tcp.rs`, `tests/server_tcp.rs`.
- **In-module dispatch** — `#[cfg(test)] mod tests` inside `src/server/dispatch.rs`
  drives a real `Dispatcher` via a `Harness::request(op, payload)` / `empty_request(op)`
  helper. This IS the wire dispatch layer (opcode→handler→`ResponseFrame`), just
  without a TCP socket. It is counted as wire-level coverage below.

Column meanings:
- **Happy path** — successful op, result asserted.
- **Error codes** — does *any* test trigger this op's applicable `ERR_*` wire codes via the dispatcher? (Applicable codes listed per op.)
- **Batch boundary** — empty batch / max-batch / over-max / partial-error (some items fail, some succeed) / large batch.
- **Crash mid-op** — kill between redo-fsync and data write / recovery replay idempotency for THIS op.
- **Repl. failure mid-op** — op issued while a replica is slow/dead/timing-out (degraded path, quorum).
- **Migration in progress** — op issued against a pending-inbound / fenced / migrating shard.
- **Single-node** — exercised on a non-clustered server.
- **Cluster** — exercised on a multi-node cluster (routing/master/redirect).

## Cross-cutting harness coverage (not per-op)

- **Crash/recovery boundary harness**: `recovery_crash_boundaries.rs` (5 boundary points), `fault_injection.rs` (redo-fsync / writev-partial / secondary-redb reconcile). Strong for **create/spend/free**; absent for freeze/unfreeze/reassign/conflicting/locked/preserve_until/set_mined/mark_longest as *named* crash cases.
- **Replication harness**: `replication_tcp.rs` exercises these `ReplicaOp` variants (verified by op-call census across the file): **Create (1×), Spend (8×), SetMined (3×), Freeze (5×), MarkLongestChain (4×), SetLocked (1×)**. The remaining six mutations — **unspend, unfreeze, reassign, set_conflicting, preserve_until, delete** — never traverse the replica apply path in any test. (The `ReplicaOp` enum's coverage of *unspend/reassign/conflicting/delete* on the replica side is therefore unverified end-to-end; a divergent replica apply for those ops would not be caught.)
- **Migration harness**: `cluster_tcp.rs` + `cluster_edge_cases.rs` are thorough on shard move / fence / redirect / no-loss / no-dup, but assert almost exclusively with **create / spend / get**; other mutations are never issued against a migrating shard.
- **e2e / property**: `e2e_workload.rs` (mixed single-node + crash-injection seeds + reorg + mempool churn), `tests/simulation`, `tests/stress`, `tests/workload` provide randomized cross-op coverage that partially backfills the per-op gaps above (but does not assert specific error codes).
