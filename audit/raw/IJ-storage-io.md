## Findings

### [HIGH] Missing external blob on the production read path is reported as TX_NOT_FOUND, masking data loss
**Location:** `src/ops/engine.rs:2631` (`Engine::read_cold_data`), error mapping at `src/server/dispatch.rs:6568`
**What's wrong:** The production cold-data read path is `Engine::read_cold_data` (dispatched at `dispatch.rs:5717`), not `StorageManager::read_cold_data`. When a record carries `TxFlags::EXTERNAL` and `blob_store.get` returns `Ok(None)`, the engine returns `SpendError::TxNotFound`, which the dispatcher maps to `ERR_TX_NOT_FOUND`. The F-G9-001 fix ("missing blob must be a typed integrity error, not indistinguishable from absence") landed only in `StorageManager::read_cold_data` (`src/storage/manager.rs:266`, `StorageError::ColdDataNotFound`) — and `StorageManager` has **zero production callers** (only tests `g9_001`/`g9_002` use it). The wire protocol even defines error 17 `BLOB_NOT_FOUND` (README, `opcodes.rs:359`), but it is only emitted on the create path (`dispatch.rs:3656,3674`), never on reads.
**Why it matters:** A lost/GC'd/bit-rotted blob for an existing, index-registered transaction is reported to Teranode as "transaction does not exist" — semantically false (the tx exists; its UTXOs are spendable). Callers may treat the tx as never-created and recreate it, or skip validation, exactly the failure mode F-G9-001 was supposed to eliminate. The fix was applied to a dead code path.
**Reproduction:** Create an external record via the dispatch path (`OP_STREAM_CHUNK`/`OP_STREAM_END` then `OP_CREATE_BATCH` with `FLAG_EXTERNAL_BLOB`), delete the blob files from `blobstore_path`, then issue the cold-data read op. Observe `ERR_TX_NOT_FOUND` instead of a blob/integrity error.
**Suggested fix:** Add a distinct `SpendError::BlobNotFound { txid }` variant in `Engine::read_cold_data` for the `Ok(None)` branch and map it to `ERR_BLOB_NOT_FOUND` (17) in dispatch. Align with `StorageError::ColdDataNotFound` semantics or delete `StorageManager` and consolidate (see dead-code finding below).

### [HIGH] Blob-GC TOCTOU: aged blob can be deleted between create's digest check and index registration
**Location:** `src/storage/blob_gc.rs:121-134` (documented contract), `src/server/dispatch.rs:3661` (digest check), `src/storage/blobstore.rs:299-316` (`list_for_gc` grace)
**What's wrong:** F-G9-004's mitigation (skip blobs younger than `PERIODIC_GC_MIN_BLOB_AGE` = 60 s) only protects blobs whose **mtime is fresh**. The create dispatch reads `bs.digest(&txid)` (proving the blob exists) and then registers the index entry; if the blob was uploaded more than 60 s before the create lands (a client may legitimately stream the blob, then send `OP_CREATE_BATCH` minutes later — nothing in the protocol forbids it; ditto a re-create of a previously deleted txid whose blob is still on disk), the blob is past the grace window. A periodic GC sweep running concurrently sees an aged blob with no index entry → deletes it. The create then completes and registers an `EXTERNAL` entry whose blob is gone: permanent cold-data loss, surfacing later as digest/not-found errors. The module's own doc admits the sweep "can mis-classify a half-completed create" but claims the grace covers it — it covers only the young-blob half of the race.
**Why it matters:** Unrecoverable cold-data loss for a successfully acknowledged create. Window is small (sweep is hourly, race window is the create dispatch), but at sustained load over months this fires.
**Reproduction:** Test sketch: `FileBlobStore::put`, backdate mtime past 60 s (as `g9_004` does), start `reconcile_orphan_blobs` on one thread while a second thread runs digest-check-then-`engine.register` with a sleep injected between them; assert blob survives iff registration is observed. Today the blob is deleted.
**Suggested fix:** Make the create path re-verify `bs.digest()`/`exists()` AFTER index registration and fail/compensate the create if the blob vanished; or maintain an in-flight-txid set (registered at digest-check time, cleared after register) that the GC sweep consults; or have GC re-check the index immediately before each delete and skip txids holding a create stripe lock.

### [MEDIUM] Tier thresholds are advisory: production tier selection is the client's FLAG_EXTERNAL_BLOB, INLINE_THRESHOLD is never enforced
**Location:** `src/ops/engine.rs:2058-2154` (create), `src/server/dispatch.rs:3651` (`is_ext = item.flags & FLAG_EXTERNAL_BLOB`), `src/storage/tiers.rs:29` (`tier_for_size` — no production caller), README:580
**What's wrong:** Checklist item "tx growing 7 KiB → 9 KiB → 1.1 MiB lands in right tier each time, or growth is rejected." Answer, with evidence: **records never change tier because they are immutable after create and the tier is chosen by the client, not by size.**
- A re-create of an existing txid is rejected (`engine.rs:2067-2069`, `CreateError::DuplicateTxId`) — there is no cold-data-update op, so "growth after create" cannot happen. ✔
- BUT the 8 KiB `INLINE_THRESHOLD` (`tiers.rs:9`) is consulted **nowhere** on the production create path. `Engine::create` writes whatever cold data the client sent inline (up to the wire cap `MAX_COLD_DATA_PER_ITEM` = 4 MiB, `opcodes.rs:539`) unless the client set `FLAG_EXTERNAL_BLOB`, in which case any size (including tiny payloads) goes external. `StorageManager`/`tier_for_size` and all the boundary tests (`tier_8192_bytes_inline`, `tier_8193_bytes_external`, etc.) exercise code production never calls.
- README:580 states "Stored inline if <8 KiB, otherwise in the external blob store" — false as a description of server behaviour; the server enforces no such threshold.
**Why it matters:** A 4 MiB inline cold payload bloats the hot NVMe region and per-record read amplification; the documented storage model and the test suite both describe behaviour the server does not have. Conversely a client error (forgetting the flag) silently produces oversized hot records instead of an error.
**Reproduction:** Send `OP_CREATE_BATCH` without `FLAG_EXTERNAL_BLOB` carrying 3 MiB of cold data; observe an inline record of `record_size ≈ 3 MiB` (`meta.record_size`, allocator stats). Send a 100-byte payload with the flag; observe an external record.
**Suggested fix:** Either enforce `tier_for_size` server-side in the create dispatch (reject or route oversized inline payloads), or update README/phase-11 docs and delete `tier_for_size`'s false promise. Pick one; don't keep both stories.

### [MEDIUM] StorageManager and BlobUploader are dead code in production — and the uploader contains two latent bugs
**Location:** `src/storage/manager.rs` (whole type), `src/storage/uploader.rs:202-273`
**What's wrong:** Grep shows no production caller of `StorageManager` (only tests `g9_001`, `g9_002`) or `BlobUploader` (only tests `g9_003`, `g9_008`; `manager.rs:184` doc points at it). Production cold-data writes go through `Engine::write_full_record_with_cold` + `BlobStore::put` directly. The uploader, if ever wired up as the docs suggest, has:
1. **Lost-update race:** `write_external_ref` (`uploader.rs:264-273`) does an unsynchronized read-modify-write of the *entire* 256-byte metadata header from the background thread (`io::read_metadata` → mutate → `io::write_metadata`) without taking the engine stripe lock or the `io_locks()` record guard. A concurrent spend/setMined on the same record between the read and the write is silently reverted (generation, spent_utxos, flags all rolled back), i.e. record corruption.
2. **Wrong field semantics:** `outputs_offset: task.inputs_len as u64` (`uploader.rs:220`) stores the input *count* into a field documented as a *byte offset* within the blob (`record.rs:293`).
**Why it matters:** Dead code that documentation actively recommends ("For async upload, use BlobUploader") is a loaded foot-gun; the first person to wire it up inherits a metadata-corruption race. Meanwhile the integration tests g9_001/g9_002/g9_003/g9_008 pin behaviour of code that cannot affect production, giving false audit confidence (see the TX_NOT_FOUND finding above for the consequence).
**Reproduction:** Code inspection suffices for the dead-code claim (`grep -rn 'StorageManager\|BlobUploader' src/` excluding storage module + tests). For the race: spawn an uploader against a `MemoryDevice`, run concurrent `engine.spend` on the same record during `process_task`, observe generation rollback.
**Suggested fix:** Delete `BlobUploader` and `StorageManager` (or fold the surviving pieces — `ColdData`, `inline_cold_offset`, the F-G9 error variants — into the engine path) and retarget g9_001/g9_002 at `Engine::read_cold_data`.

### [MEDIUM] Unconfigured blob store silently degrades EXTERNAL records to empty cold data
**Location:** `src/ops/engine.rs:2607-2647`
**What's wrong:** `read_cold_data` enters the external branch only when the flag is set **and** `self.blob_store` is `Some`. With `blob_store = None`, an EXTERNAL record falls through to the inline branch; `record_size == metadata + slots` so `cold_size == 0` and it returns `Ok(vec![])` — exactly the pre-F-G9-001 "silent empty" bug, resurrected for the unconfigured case. `bin/server.rs:606` always configures a store, but `Engine` is a public API and several constructions (recovery paths, tests, embedded use) build engines without one.
**Why it matters:** Indistinguishable "no cold data" vs "cold data exists but unreachable" — for validation/SPV that is a correctness hazard, and it is silent (no log, no error).
**Reproduction:** Build an `Engine` without `set_blob_store`, register an EXTERNAL record, call `read_cold_data`; observe `Ok(vec![])`.
**Suggested fix:** In `read_cold_data`, when `EXTERNAL` is set and `blob_store` is `None`, return an error (`StorageError`/`BlobNotFound`-equivalent) instead of falling through.

### [MEDIUM] README documents an io_uring scaffolding module (`src/device_io/`) that does not exist
**Location:** `README.md:26` ("The `io_uring` backend in `src/device_io/` is scaffolding only…"), `README.md:700` (project tree lists `device_io/`); contrast `src/lib.rs:10` ("io_uring/sync fallback backends … deleted on 2026-05-28") and `ls src/` (no such directory)
**What's wrong:** The module was removed 2026-05-28 (lib.rs comment, CLAUDE.md), but the README still claims it exists as scaffolding. Checklist disposition: the code path that actually runs in production is synchronous positional `pread`/`pwrite` via `DirectDevice` (O_DIRECT on Linux); there are **no** io_uring remnants in `src/` or `Cargo.toml` — only the README misleads.
**Why it matters:** An operator or reviewer reading the README will look for, and reason about, an I/O backend that isn't there; stale architecture docs are how the next misdesign starts.
**Reproduction:** `grep -rn io_uring src Cargo.toml README.md`; `ls src/device_io` → no such directory.
**Suggested fix:** Delete README lines 26 and the `device_io/` tree entry; state plainly that I/O is synchronous O_DIRECT `pread`/`pwrite` (matches `src/device.rs`).

### [MEDIUM] Allocator `free()` has no double-free / overlap detection — corrupts the freelist silently in release builds
**Location:** `src/allocator.rs:609-685` (`free`), `src/allocator.rs:320-334` (`FreelistBackend::insert`), `src/allocator.rs:361-371` (`debug_assert_sorted`)
**What's wrong:** `free()` validates only bounds (`offset >= data_region_start`, `end <= device_size`, non-zero size). Freeing a region that is already free (or overlaps an existing free region) inserts a duplicate/overlapping `FreeRegion`. In debug builds `debug_assert_sorted` fires only for the *exact-duplicate-offset* case (strict `<` ordering); overlapping-but-distinct offsets pass the assert entirely. In release builds both cases silently corrupt the freelist → the same device range handed out to two different records → data corruption. The crash-replay path (`replay_free`) has an idempotency guard ("fully covered by existing free region → no-op"), the runtime path has none — and `is_allocated_range()` exists (`allocator.rs:1296`) but is not consulted by `free()`. Callers (engine `delete`, create-rollback `release_create_reservation` in `dispatch.rs:3589`) are believed single-free today, so this is hardening, not an observed bug.
**Why it matters:** One bug anywhere in the rollback/delete plumbing (e.g. a double rollback after a partially failed batch) converts into silent cross-record corruption, the worst failure class in this system.
**Reproduction:** Unit test: `alloc.allocate(8192)`, `alloc.free(off, 8192)`, `alloc.free(off, 8192)` (release build) → freelist contains the region twice; two subsequent `allocate(8192)` calls return the same offset.
**Suggested fix:** In `free()`, reject (typed `InvalidFree`/`DoubleFree` error) when the range intersects an existing free region — `free_region_containing`/`prev_before`+`next_from` already provide the lookups at O(log n).

### [LOW] Engine::delete leaves external blobs on disk until the next hourly GC sweep
**Location:** `src/ops/engine.rs:3960-4028` (`delete` — no blob deletion), `src/storage/manager.rs:374-380` (`delete_cold_data` — dead code)
**What's wrong:** Deleting an EXTERNAL record tombstones the device record and unregisters the index entry but never calls `blob_store.delete`. The blob becomes an orphan reclaimed by the periodic sweep (default interval 1 h) after the 60 s grace. `StorageManager::delete_cold_data` implements eager deletion but has no production caller.
**Why it matters:** Deferred reclamation is a defensible design (GC is the single owner of blob lifetime, simplifying crash-consistency), but it is undocumented at the delete site, and bursts of large-tx deletions can hold gigabytes for up to an hour plus grace.
**Reproduction:** Create external record, delete it via OP_DELETE, observe blob still in `blobstore_path` until `reconcile_orphan_blobs` runs.
**Suggested fix:** Either document the deferred-reclamation contract on `Engine::delete`, or best-effort `blob_store.delete` after index unregistration (GC remains the backstop).

### [LOW] `FileBlobStore::digest()` skips the per-key lock that F-G9-005 added to the other read paths
**Location:** `src/storage/blobstore.rs:758-769`; contrast `get` (:707), `get_range` (:717), `stream_to` (:780)
**What's wrong:** `digest()` reads `path.exists()`, `meta_path.exists()`, then `read_meta` without holding `key_locks[..]`. A concurrent `FileStreamWriter::finish` (payload rename, then sidecar write, under the lock) can interleave so `digest()` returns the *old* sidecar while the *new* payload is already in place. The create dispatch stamps that stale digest into `ExternalRef` (`dispatch.rs:3661`) → every subsequent read fails the record-anchored hash check. In practice both uploads for the same txid should carry identical bytes (content is a deterministic function of the tx), so impact is theoretical — which is why this is LOW, not HIGH.
**Reproduction:** Two threads: one loops `begin_stream/finish` with payload A then B for key K; other loops `digest(K)`+`get(K)` asserting the digest matches the payload read. Occasional mismatch without the lock.
**Suggested fix:** Take `self.key_locks[Self::lock_index(key)]` in `digest()` (one-liner, matches the F-G9-005 pattern).

### [LOW] macOS `F_NOCACHE` fcntl result is ignored
**Location:** `src/device.rs:884-892`
**What's wrong:** `libc::fcntl(fd, F_NOCACHE, 1)` return value is discarded. If it fails, macOS development I/O silently runs fully page-cached, diverging further from Linux O_DIRECT behaviour. macOS is a dev-only platform here (Linux is the production target, where O_DIRECT is set at open and its failure is surfaced by `opts.open(path)?`), so impact is dev-fidelity only.
**Reproduction:** Inspection; or pass an fd type that rejects F_NOCACHE and observe no error.
**Suggested fix:** Check the return and `tracing::warn!` on failure.

### [LOW] Linux block-device size-query branch has no integration coverage; macOS test skips silently
**Location:** `tests/block_device_size.rs` (`#![cfg(target_os = "macos")]`), `src/device.rs:828-845`
**What's wrong:** Judging the test (checklist item): it is well built — real `S_IFBLK` node via unprivileged `hdiutil ram://`, proves the bogus 4096-byte size hint is ignored, asserts `size() == block_count*block_size`, exercises the OOB bounds check and a real positional round-trip, RAII detach. Two gaps: (1) it compiles only on macOS, so the **Linux** `BLKGETSIZE64` branch — the production one — is covered only by the pure helper `validate_block_device_size` unit test, never against a real/loop block device; (2) when `hdiutil` is unavailable the test returns early, so CI may report green without the body ever running (project rules ban `#[ignore]`, but a silent early-return is the same coverage hole wearing a different hat).
**Reproduction:** Run the suite on Linux: no block-device integration test exists. Run on a macOS sandbox without `hdiutil`: test passes having asserted nothing.
**Suggested fix:** Add a Linux twin using `losetup` on a temp file (skip-with-message when not root/capable), and emit the skip as a test-visible artifact (e.g. a separate `#[test]` that asserts the capability and is allowed to fail loudly in environments that promise it).

## Checklist disposition

### I. Storage tiers and blobs
- **Threshold transitions (7 KiB → 9 KiB → 1.1 MiB):** ⚠️ — Growth-after-create is structurally impossible: records are immutable post-create and duplicate txids are rejected (`engine.rs:2067`, `CreateError::DuplicateTxId`); no cold-data-update opcode exists. BUT the size thresholds themselves are not enforced server-side — tier = client's `FLAG_EXTERNAL_BLOB`; `INLINE_THRESHOLD`/`tier_for_size` are production-dead (MEDIUM finding). A 9 KiB or 1.1 MiB payload lands wherever the client says, up to the 4 MiB wire cap.
- **Missing blob → BLOB_NOT_FOUND:** ❌ — On the production read path a missing blob surfaces as `ERR_TX_NOT_FOUND` (`engine.rs:2631` → `dispatch.rs:6568`); `BLOB_NOT_FOUND` (17) is emitted only at create time. HIGH finding. (Create-time check itself: ✅ `dispatch.rs:3653-3687`.)
- **Orphaned blobs eventually cleaned / GC-vs-create race:** ✅ for the leak (recovery sweep + hourly periodic sweep, `blob_gc.rs`, tests `blob_gc_recovery.rs`); ⚠️ for the race — g9_004's grace covers young blobs only; aged-blob TOCTOU between digest-check and register remains (HIGH finding).
- **Concurrent uploads to same txid:** ✅ — per-key mutex (256 stripes, `blobstore.rs:328,677,618`), unique per-process tmp names (`unique_tmp_path`), atomic rename; `finish` holds the key lock across payload-rename + sidecar write. Residual: `digest()` skips the lock (LOW finding).
- **blobstore_path not writable / disk full:** ✅ — all failures propagate as `BlobError::Io` (no unwrap/expect in `blobstore.rs` write paths); stream write failure → abort/Drop removes `.tmp` (g9_007); dispatch maps to `ERR_STORAGE_IO`. Minor: `bin/server.rs:606` never validates the path at startup, so a typo'd path is discovered at first external op, not at boot.
- **Blob content integrity verified on read:** ✅ — two layers: sidecar digest verified by `FileBlobStore::get/get_range/stream_to` (`read_and_verify`, two-pass `stream_to`), and record-anchored `ExternalRef.content_hash` + length cross-check on the engine read path (`engine.rs:2613-2628`). g9_002 judged correct but pinned to the dead `StorageManager` path (see judgements).

### J. I/O layer
- **Direct I/O triple alignment (offset, length, buffer address):** ✅ — enforced before every syscall by `check_alignment_impl` (`device.rs:190-215`) on both `DirectDevice` and `MemoryDevice` (so CI on MemoryDevice catches violations that would EINVAL on Linux). Alignment is guaranteed **by construction**: every production call site audited (`io.rs`, `storage/manager.rs`, `storage/input_refs.rs`, `index/`, `redo.rs`, `recovery.rs`, `ops/engine.rs`) allocates via `AlignedBuf::new(len, device.alignment())`, rounds offsets to `aligned_base`, and slices buffers only from index 0 (`&buf[..n]`, address-preserving). No raw `Vec` reaches `pread`/`pwrite` outside tests. Unit tests cover misaligned buffer-address rejection (`direct_device_rejects_misaligned_buffer_address`).
- **Block device size from kernel, config ignored:** ✅ — `BLKGETSIZE64` (Linux) / `DKIOCGETBLOCKCOUNT×DKIOCGETBLOCKSIZE` (macOS) via compile-time-encoded `nix::ioctl_read!`; geometry validated (zero size, u64 overflow → `InvalidBlockDeviceGeometry`); `set_len` never called on block devices; `size` param documented as regular-file-only. Test judged good but macOS-only with silent capability skip; Linux branch lacks integration coverage (LOW finding).
- **Regular file grown, never truncated:** ✅ — `OpenOptions.truncate(false)`, `set_len` only when `existing < size` (`device.rs:873-882`); pinned by `direct_device_no_truncate_existing` and `direct_device_grows_existing_file`.
- **Partial pread/pwrite retried, EINTR handled:** ✅ — `pread_exact_at`/`pwrite_all_at` loop with checked continuation offsets, typed `ShortRead` on EOF and `WriteStalled` on zero progress (`device.rs:310-385`); EINTR retried inside `DirectDevice::pread`/`pwrite` (`device.rs:950-957, 1002-1009`); short-read/short-write loops unit-tested via a chunked fake device. Production callers use only the exact-at variants (bare `.pread(`/`.pwrite(` call sites are all in `device.rs` itself).
- **io_uring status:** ✅ code / ❌ docs — no io_uring code or dependency remains; production path is synchronous O_DIRECT positional I/O. README still advertises a nonexistent `src/device_io/` scaffolding module (MEDIUM finding).
- **macOS F_NOCACHE vs Linux O_DIRECT:** ⚠️ — Linux gets real O_DIRECT at open (failure surfaced); macOS gets advisory F_NOCACHE with the fcntl result ignored (LOW finding). MemoryDevice deliberately enforces O_DIRECT-style rules so the dev/CI gap doesn't hide alignment bugs.
- **Allocator double-free / leak:** ⚠️ — overflow alloc accounting leak fixed and well-tested (g2 test judged good); runtime `free()` has no double-free/overlap rejection — silent freelist corruption in release on a caller bug (MEDIUM finding). Redo-replay paths ARE idempotent-guarded.
- **Device write errors (ENOSPC, EIO) surfaced:** ✅ — propagate as `DeviceError::Io` through `pwrite_all_at`; engine maps to `SpendError::StorageError`/`CreateError` → `ERR_STORAGE_IO`; redo/allocator paths roll back reservations on flush failure. One lossy spot: `dispatch.rs:4945` collapses blob `get` errors to `None` via `.ok().flatten()` in the delete-snapshot path (still fails the op with `ERR_STORAGE_IO`, but the underlying error text is discarded).
